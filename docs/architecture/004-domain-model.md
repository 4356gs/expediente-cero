# Domain Model

## Purpose

Define the stable business concepts used by the API, persistence layer, model
schemas, and reviewer interface without coupling them to any framework.

## Aggregate root: Case

`Case` owns the intake lifecycle and is the only aggregate that may change
case status.

Required attributes:

- `id`: UUID generated by the application;
- `reference`: short human-readable synthetic reference;
- `procedure_type`: selected or confirmed supported procedure;
- `output_language`: Spanish or Galician;
- `status`: current lifecycle state;
- `created_at`: timezone-aware UTC timestamp;
- `updated_at`: timezone-aware UTC timestamp.
- `validation_completed_at`: set only after all deterministic results are
  persisted atomically;
- `validation_template_version`: version of the rules that produced the current
  deterministic results.

Relationships:

- one or more source messages;
- zero or more document metadata records;
- zero or more model runs;
- zero or one current intake analysis;
- zero or more validation findings;
- zero or more checklist results;
- zero or one current follow-up draft;
- zero or one final review decision;
- one or more audit events.

## Supporting entities

Represents the unstructured synthetic request supplied by the reviewer.

- `id`
- `case_id`
- `content`
- `is_synthetic`
- `created_at`

`is_synthetic` must be true in the hackathon MVP.

### DocumentMetadata

Describes a synthetic document without storing file bytes.

- `id`
- `case_id`
- `document_type`
- `display_name`
- `is_synthetic`
- `created_at`

### IntakeAnalysis

Typed result returned from the model and accepted by schema validation.

- `id`
- `case_id`
- `procedure_type`
- `procedure_reason`
- `facts`
- `assumptions`
- `unresolved_questions`
- `contradictions`
- `requested_output_language`
- `prompt_version`
- `model_run_id`

### ExtractedFact

- `source_reference`

- `item_code`

- `required`

### ValidationFinding

- `id`

- `model_text`
- `reviewed_text`
- `prompt_version`
- `model_run_id`
- `created_at`

### ReviewDecision

- `id`
- `case_id`
- `decision`: `approved` or `rejected`
- `reason`
- `reviewer_label`
- `created_at`

Rejection requires a reason. Approval is forbidden while blocking findings
remain active.

### ModelRun

- `id`
- `case_id`
- `purpose`: `intake_analysis` or `follow_up_draft`
- `provider`
- `model`
- `prompt_version`
- `started_at`
- `completed_at`
- `status`: `succeeded`, `failed`, or `refused`

- `sanitized_error_code`

Prompts, API keys, hidden reasoning, and raw sensitive content are not stored in

### AuditEvent

- `id`
- `case_id`
- `event_type`
- `actor_type`
- `actor_label`
- `recorded_at`
- `sanitized_metadata`

Audit events are append-only.

## State transitions

Allowed transitions:

- `draft` -> `analyzing`
- `analyzing` -> `analyzed`
- `analyzing` -> `analysis_failed`
- `analysis_failed` -> `analyzing`
- `analyzed` -> `needs_review`
- `needs_review` -> `analyzing`
- `needs_review` -> `approved`
- `needs_review` -> `rejected`

`approved` and `rejected` are terminal in the MVP.

## Aggregate invariants

1. A case cannot be approved without an intake analysis.
2. A case cannot be approved without deterministic validation results.
3. A case cannot be approved while any blocking finding is active.
4. A rejected case must include a human-provided reason.
5. Model output cannot directly transition a case to `approved` or `rejected`.
6. Every state transition must create an audit event atomically.
7. Every model-derived record must reference its model run and prompt version.
8. All hackathon source messages and document metadata must be synthetic.
9. Original model text is immutable after persistence.
10. Only the application service may coordinate aggregate state transitions.
11. An `analyzed` case has a persisted intake analysis but no implied deterministic
    validation result.

## Procedure templates

Each procedure template defines:

- supported fact fields;
- required fact fields;
- required document metadata types;
- deterministic consistency rules;
- follow-up message context;
- stable checklist item codes.

Templates are versioned application configuration. They do not encode legal
eligibility or replace professional review.

The initial version is `deterministic-validation-v1`. Its exact required facts
and document types are defined in the Build Week implementation plan. Only
non-blank stated facts with valid persisted evidence satisfy required fields.
A completed validation may contain blocking findings while the case is in
`needs_review`.
